# ClaudeMaster 产品设计文档

## 1. 产品定位

### 一句话描述

ClaudeMaster 是一个让开发者在手机或电脑浏览器上**监督、审查和指挥**多个 Claude Code AI 编程代理的管理平台。

### 核心隐喻

把每个 Claude Code 实例想象成一个**远程工作的初级程序员**。它能力很强，但会犯错。你是这些程序员的技术主管——你需要：

- 随时知道每个人在做什么
- 审批它们的工作计划再让它们动手
- 检查它们写的代码再决定是否采纳
- 在它们卡住或跑偏时及时介入
- 有时候同时管好几个人

ClaudeMaster 就是你的**管理面板**。

### 使用场景

**场景 A：办公桌前多线程工作**
> 你在工作站上同时开了 3 个 Claude Code，分别在做不同的功能。你在一个浏览器标签页里看到三个代理的状态，哪个在等你审批计划，哪个写完代码等你 review，哪个还在跑——一目了然。

**场景 B：离开电脑后用手机监控**
> 你给 Claude Code 安排了一个大任务就去吃饭了。手机上打开 ClaudeMaster，看到它进入了 Plan 模式在等你审批。你在手机上读完计划，点了批准，它继续干活。吃完饭回来，代码已经写好在等你 review。

**场景 C：手机上快速介入**
> 你在沙发上用手机浏览之前的对话历史，发现某个 Claude Code 似乎理解错了需求。你直接在手机上输入一条修正指令发过去。

## 2. 用户角色与核心诉求

本产品只有一个用户角色：**开发者本人**（个人工具，无多用户需求）。

| 诉求 | 优先级 | 说明 |
|------|--------|------|
| 知道每个代理在干什么 | P0 | 不打开终端就能掌握全局 |
| 审阅代理的工作计划 | P0 | Plan 模式是 AI 犯错的第一道防线 |
| 审查代理修改的代码 | P0 | Code Review 是第二道防线 |
| 从手机上完成以上操作 | P0 | 移动端不是"凑合看"，是主要使用方式之一 |
| 阅读历史对话 | P1 | 回顾之前的上下文、学习 AI 的做法 |
| 向代理发送指令 | P1 | 文字输入，不需要完整终端 |
| 处理代理的权限请求 | P1 | AI 要跑危险命令时需要你批准 |
| 启动/停止/恢复会话 | P2 | 基本进程管理 |
| 搜索历史对话 | P2 | 跨项目全文检索 |
| 用量与费用统计 | P3 | Token 消耗追踪 |

## 3. 信息架构

```
ClaudeMaster
├── 工作台（首页）                ← 核心页面：所有代理的实时状态
│   ├── 需要关注的代理（置顶）     ← 等待审批、出错、被阻塞的
│   ├── 正在工作的代理            ← 活跃运行中的
│   └── 空闲/已完成的代理         ← 最近结束的
│
├── 对话查看器                   ← 阅读完整对话
│   ├── 消息流                  ← 用户/AI 消息、工具调用、思维过程
│   ├── 代码变更                ← 本次会话的所有文件修改（diff 视图）
│   └── 操作栏                  ← 发送指令、审批操作
│
├── 会话历史                    ← 浏览所有历史会话
│   ├── 按项目分组
│   ├── 按时间排序
│   └── 搜索与筛选
│
└── 设置                       ← 通知偏好、认证令牌、主题
```

## 4. 核心交互流程

### 4.1 代理状态机

每个 Claude Code 代理在 ClaudeMaster 中有以下状态：

```
  ┌──────────┐
  │  未启动   │ ← 历史会话，可恢复
  └────┬─────┘
       │ 启动/恢复
       ▼
  ┌──────────┐     提交计划      ┌──────────────┐
  │  工作中   │ ──────────────→  │  等待计划审批  │
  │          │ ←────────────── │              │
  └────┬─────┘   批准/驳回       └──────────────┘
       │
       │ 请求权限
       ▼
  ┌──────────────┐
  │  等待权限批准  │ ← AI 要执行危险操作
  └────┬─────────┘
       │ 批准/拒绝
       ▼
  ┌──────────┐     写完代码      ┌──────────────┐
  │  工作中   │ ──────────────→ │  等待代码审查  │
  │          │ ←────────────── │              │
  └────┬─────┘   通过/修改       └──────────────┘
       │
       │ 任务完成 / 出错 / 空闲
       ▼
  ┌──────────┐
  │ 已完成/出错│
  └──────────┘
```

**关键设计：「需要关注」优先**

在工作台上，**最重要的信息永远是"哪些代理在等你"**。等待审批的代理应该：
- 置顶显示，视觉突出（例如黄色边框）
- 一键可达审批界面
- 手机上能直接操作，不需要缩放或横向滚动

### 4.2 计划审批流程

这是防止 AI 犯错的**最核心交互**。

```
AI 进入 Plan 模式
       │
       ▼
┌─────────────────────────────────┐
│         计划审批卡片              │
│                                 │
│  📋 计划标题（AI 生成的摘要）     │
│                                 │
│  计划正文（Markdown 渲染）        │
│  - 步骤 1：修改 xxx.py 的 ...    │
│  - 步骤 2：新建 yyy.ts ...       │
│  - 步骤 3：运行测试 ...           │
│                                 │
│  涉及文件：                      │
│  [xxx.py] [yyy.ts] [zzz.test]   │
│                                 │
│  ┌────────┐ ┌────────┐ ┌─────┐ │
│  │  批准 ✓ │ │ 驳回 ✗ │ │ 备注 │ │
│  └────────┘ └────────┘ └─────┘ │
└─────────────────────────────────┘
```

**移动端**：计划卡片全宽显示。底部固定操作按钮，拇指可达。

**批准**：向 Claude Code 发送继续执行的信号。
**驳回**：弹出文本框让你写明原因，AI 收到后调整计划。
**备注**：不批准也不驳回，只是追加一条补充说明给 AI。

### 4.3 代码审查流程

AI 修改完代码后，用户需要检查改动。

```
┌─────────────────────────────────┐
│          代码审查视图             │
│                                 │
│  变更摘要：修改了 3 个文件        │
│                                 │
│  ▸ backend/services/parser.py   │  ← 点击展开 diff
│    +12 行  -3 行                │
│                                 │
│  ▾ frontend/src/viewer.ts       │  ← 已展开
│  ┌─────────────────────────────┐│
│  │ - const old = "foo";        ││
│  │ + const new = "bar";        ││
│  │   // unchanged line         ││
│  │ + added line                ││
│  └─────────────────────────────┘│
│                                 │
│  ▸ tests/parser_test.py        │
│    +25 行  -0 行                │
│                                 │
│  ┌──────┐ ┌──────┐ ┌────────┐  │
│  │ 采纳  │ │ 回退  │ │ 要求修改│  │
│  └──────┘ └──────┘ └────────┘  │
└─────────────────────────────────┘
```

**数据来源**：
- 对比会话开始时的 git 状态与当前工作树（`git diff`）
- 或利用 Claude Code 自带的 file-history（`~/.claude/file-history/`）逐文件追踪

**移动端 diff 阅读**：
- 默认使用**统一 diff 视图**（unified diff），不使用左右对比
- 代码块支持横向滑动
- 文件列表可折叠，避免一次性加载过多内容
- 字体大小适配移动屏幕（可调节）

### 4.4 向代理发送指令

用户在任何时候都可以向某个 Claude Code 代理发送文字指令。

```
┌─────────────────────────────────┐
│  ┌─────────────────────────┐    │
│  │ 请把那个函数改成异步的     │    │
│  └─────────────────────────┘    │
│           ┌──────┐              │
│           │ 发送  │              │
│           └──────┘              │
└─────────────────────────────────┘
```

**实现方式**：
- 通过 Claude Code 的 `--input-format stream-json` 管道注入用户消息
- 或写入一个约定的文件由 Claude Code 读取
- 最简方案：在对话查看器中显示"该功能需要在终端中操作"，提供快速复制命令

**MVP 阶段可以先不实现实时交互**，只提供"浏览对话 + 审查代码"的只读能力。指令发送作为第二阶段目标。

### 4.5 权限请求处理

Claude Code 请求执行危险操作时（如 `rm -rf`、`git push --force`），用户需要审批。

```
┌─────────────────────────────────┐
│  ⚠️ 权限请求                     │
│                                 │
│  Claude Code 想要执行：          │
│  ┌─────────────────────────────┐│
│  │ git push origin main        ││
│  └─────────────────────────────┘│
│                                 │
│  上下文：刚完成了 feature-x 的   │
│  开发，想推送到远程仓库           │
│                                 │
│  ┌────────┐    ┌────────┐      │
│  │  批准   │    │  拒绝   │      │
│  └────────┘    └────────┘      │
└─────────────────────────────────┘
```

**注意**：这个功能依赖于能与 Claude Code 进程双向通信。在 MVP 阶段，权限请求仍然在终端中处理，ClaudeMaster 仅展示对话中出现的权限请求记录。

### 5.2 对话查看器

这是**信息密度最高**的页面。需要在可读性和信息完整性之间取得平衡。

**消息类型与展示策略**：

| 消息类型 | 展示方式 | 默认状态 |
|---------|---------|---------|
| 用户文字消息 | 全文显示，浅色背景 | 展开 |
| AI 文字回复 | Markdown 渲染，代码高亮 | 展开 |
| AI 思维过程 (thinking) | 灰色斜体，可折叠 | **折叠** |
| 工具调用 (tool_use) | 工具名标签 + 输入/输出 | **折叠**，仅显示工具名 |
| 工具结果 (tool_result) | 合并到对应工具调用中 | 跟随工具调用 |
| 系统消息 | 小字提示条 | 内联 |

**折叠策略的关键**：对话查看器的默认折叠状态要让用户**快速抓住主干**。thinking 和 tool_use 是辅助细节，默认折叠；文字消息是对话主线，默认展开。

**快捷操作**：
- 「全部展开」/「全部折叠」按钮
- 「只看对话」模式——隐藏所有工具调用，只看 user/assistant 文字
- 「跳到底部」——大对话中快速定位最新消息

## 6. AI 犯错时的防线体系

这是本产品的**核心价值主张**——帮助用户在 AI 工作流中保持控制力。

### 第一道防线：计划审批

**何时触发**：Claude Code 进入 Plan 模式时。
**用户能做什么**：阅读计划 → 批准 / 驳回并说明原因 / 追加补充说明。
**价值**：在 AI 动手写代码之前就纠正方向，成本最低。

### 第二道防线：实时对话监控

**何时触发**：AI 正在工作时。
**用户能做什么**：随时查看 AI 正在做什么，发现跑偏及时干预。
**价值**：在错误扩散之前发现问题。

### 第三道防线：代码审查

**何时触发**：AI 完成一轮修改后。
**用户能做什么**：查看所有文件变更 → 采纳 / 回退 / 要求修改。
**价值**：确保最终进入代码库的代码符合预期。

### 第四道防线：权限控制

**何时触发**：AI 要执行危险操作时。
**用户能做什么**：审查命令 → 批准 / 拒绝。
**价值**：防止不可逆的破坏性操作。

### 防线之间的关系

```
     计划审批          实时监控          代码审查        权限控制
    （方向对不对）    （过程有没有跑偏）   （结果对不对）   （操作安全吗）
        │                │                │              │
        ▼                ▼                ▼              ▼
   成本最低 ◄──────────────────────────────────────► 成本最高
   纠错越早越好                                    但永远需要最后一道防线
```

## 7. 移动端设计要点

移动端不是桌面端的缩小版，而是一个**独立的使用场景**：用户不在电脑前，需要快速监控和审批。

### 设计原则

1. **拇指优先**：所有操作按钮在屏幕底部 1/3 区域内，单手可达
2. **信息密度适中**：比桌面端少显示信息，但关键信息一个不少
3. **快速审批**：从打开 App 到完成一次计划审批 ≤ 3 次点击
4. **离线可读**：已加载的对话历史在断网后仍可浏览

### 关键交互

**快速审批流程**（从通知到完成）：
```
收到通知"MyProject 等待计划审批"
       │
       ▼ 点击通知
┌─────────────────────┐
│ 计划审批             │
│                     │
│ 计划内容...          │
│ （可滚动阅读）       │
│                     │
│                     │
├─────────────────────┤
│  [驳回]    [批准 ✓]  │  ← 底部固定，拇指可达
└─────────────────────┘
```

**对话浏览**：
- 消息气泡全宽显示，无边距浪费
- 代码块自动换行 + 支持横向滑动
- 工具调用默认折叠为一行标签（点击展开）
- 底部悬浮按钮「跳到最新」

**代码审查**：
- unified diff 视图，不支持 side-by-side（屏幕太窄）
- 文件列表可折叠
- 添加/删除行用颜色区分（绿色/红色），粗体行号
- 支持双指缩放代码区域

## 8. 通知机制

### 需要通知用户的事件

| 事件 | 紧急程度 | 通知方式 |
|------|---------|---------|
| 代理等待计划审批 | 高 | 页面内提示 + 浏览器通知 |
| 代理等待权限批准 | 高 | 页面内提示 + 浏览器通知 |
| 代理完成任务 | 中 | 页面内提示 |
| 代理遇到错误 | 中 | 页面内提示 + 浏览器通知 |
| 代理等待代码审查 | 中 | 页面内提示 |
| 新消息（订阅的会话） | 低 | 页面内实时更新 |

### 实现方式

- **页面内通知**：工作台上的状态卡片实时更新（WebSocket）
- **浏览器通知**：使用 Web Notification API，用户授权后推送桌面/手机通知
- **未来扩展**：可接入 Webhook 推送到企业微信/Telegram 等

## 9. 与 Claude Code 的交互边界

### 能做的（基于现有能力）

| 功能 | 实现方式 | 阶段 |
|------|---------|------|
| 读取对话历史 | 解析 `~/.claude/projects/**/*.jsonl` | MVP |
| 检测运行中的进程 | 扫描 `/proc/*/cmdline` | MVP |
| 实时对话流 | `watchdog` 监听 JSONL 文件追加 | 第二阶段 |
| 检测代理状态（等待审批等） | 解析最新消息中的 `tool_use` 类型 | 第二阶段 |
| 获取代码变更 | 对会话所在项目执行 `git diff` | 第二阶段 |
| 发送用户指令 | 写入 stdin 管道（需研究可行性） | 第三阶段 |
| 审批计划/权限 | 模拟终端输入（需研究可行性） | 第三阶段 |

### 暂时做不到的（Claude Code 的限制）

- **Claude Code 没有 HTTP API**——每个实例是独立的 Node.js 进程，通过 stdin/stdout 与终端通信
- **没有官方 IPC 机制**——无 Unix socket、无 TCP 端口
- **审批操作本质上是终端输入**——Plan 模式的"批准"就是用户在终端里按回车

### 渐进式策略

**MVP 阶段**：纯只读。浏览对话、查看 diff、看到代理状态。审批操作仍需回到终端。这已经解决了"手机上看不到 AI 在做什么"的核心痛点。

**第二阶段**：研究通过 PTY 注入输入的可行性。如果可行，支持从 Web 发送简单审批指令。

**第三阶段**：如果双向通信成熟，提供完整的远程终端能力（类似 web terminal）。

## 10. 技术约束与产品取舍

| 约束 | 对产品的影响 | 取舍 |
|------|------------|------|
| Claude Code 无 HTTP API | 无法实时获取精确状态 | 通过解析 JSONL 尾部推断状态 |
| JSONL 可能很大 | 对话查看器不能一次加载全部 | 分页 + 虚拟滚动 |
| 手机网络可能不稳定 | WebSocket 可能断连 | 自动重连 + 断线提示 |
| 个人工具无需复杂权限 | 安全模型可以简化 | 单 token 认证即可 |
| AI 状态推断可能不准 | 可能误判"等待审批" | 提供手动刷新，状态附带置信度提示 |

## 11. 开发优先级

### MVP（第一个可用版本）

只做**读取和展示**，不做任何写入/控制操作。

核心交付：
1. ✅ 工作台页面——看到所有代理的状态卡片
2. ✅ 对话查看器——阅读完整对话，thinking 和 tool_use 可折叠
3. ✅ 代码变更视图——查看当前会话的 git diff
4. ✅ 会话历史——浏览和搜索历史对话
5. ✅ 移动端适配——手机浏览器可用
6. ✅ 局域网访问——设置 token 后手机可访问

验收标准：**在手机浏览器上打开 ClaudeMaster，3 秒内看到所有代理的状态，点击任意一个能阅读完整对话。**

### 后续迭代

按用户价值排序，每个迭代独立可交付：
1. 实时对话流（WebSocket 推送新消息）
2. 浏览器通知（代理状态变化时推送）
3. 代理状态精确识别（等待审批 / 等待权限 / 工作中）
4. 指令发送（向代理输入文字）
5. 审批操作（批准/驳回计划）
6. 进程控制（启动/停止/恢复）
